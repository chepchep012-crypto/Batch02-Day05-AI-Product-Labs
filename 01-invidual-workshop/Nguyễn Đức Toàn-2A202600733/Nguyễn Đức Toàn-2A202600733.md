## 1. Chọn một sản phẩm để dùng thử

| Sản phẩm | AI feature | Cách truy cập |
|---|---|---|
| **V-App — V-AI** ✅ | Trợ lý voice/text, gợi ý theo ngữ cảnh trong siêu ứng dụng | App V-App (VinSmart Future — Vingroup) |

**Mô tả:** V-AI là trợ lý ảo tích hợp trong hệ sinh thái V-App của Vingroup — kết nối các dịch vụ Vinhomes, Vinpearl, VinFast, VinMart trong cùng một siêu ứng dụng. V-AI được quảng bá là có khả năng hiểu ngữ cảnh, xử lý yêu cầu đa dịch vụ và thực hiện action thay mặt user.

## 2. Dùng thử: promise vs reality

**Product hứa gì?**  
Trợ lý thông minh hiểu context của user trong hệ sinh thái Vingroup — đặt xe, quản lý booking, tư vấn dịch vụ, thực hiện giao dịch ngay trong 1 ứng dụng.

**User nào được hứa sẽ được giúp?**  
Cư dân Vinhomes, khách hàng Vinpearl, người dùng VinFast — tức là toàn bộ user trong hệ sinh thái Vingroup cần quản lý và thực hiện giao dịch cross-service.

**Kỳ vọng AI làm được task nào?**
- Tra cứu số dư, thông tin tài khoản cá nhân
- Đặt xe theo route cụ thể với giá chính xác
- Xem và xử lý booking thật (huỷ, đổi phòng, tính hoàn tiền)

**Điểm gãy quan sát được:**

### Query 1: Hỏi thông tin cá nhân
> **"Số dư ví t còn bao nhiêu?"**

| | |
|---|---|
| **Kỳ vọng** | Trả lời ngay với số dư thật |
| **Thực tế** | V-AI từ chối vì không có quyền truy cập tài khoản cá nhân, hướng dẫn user tự vào app kiểm tra |
| **Điểm gãy** | ❌ Failure — không có dữ liệu thật, không có action |

### Query 2: Cross-service request
> **"T đang ở Vinhomes, muốn đặt xe VinFast đi Vinpearl Phú Quốc, rẻ nhất bao nhiêu?"**

| | |
|---|---|
| **Kỳ vọng** | Tổng hợp xe + giá theo route thật → đặt luôn trong 1 flow |
| **Thực tế** | Trả về bảng giá cước Green SM chung (không theo route cụ thể), thông tin xe buýt miễn phí Vinpearl, 3 quick button generic |
| **Điểm gãy** | ⚠️ Low-confidence → generic info → không có booking action |

### Query 3: Multi-intent + action có hậu quả thật
> **"T vừa book Vinpearl Phú Quốc tuần tới, giờ muốn huỷ 1 đêm đầu và đổi phòng lớn hơn, còn hoàn tiền không?"**

| | |
|---|---|
| **Kỳ vọng** | Xem đơn thật → tính hoàn tiền chính xác → thực hiện huỷ/đổi ngay |
| **Thực tế** | Trả về chính sách hoàn tiền chung chung (50% trong 7 ngày), yêu cầu user tự liên hệ hotline/email, chờ 24–48h |
| **Điểm gãy** | 🔴 Action gap + Time pressure — biết policy nhưng không làm được gì |

## 3. Vẽ 4 paths

| Path | Quan sát thực tế trong V-AI |
|---|---|
| **Happy** | Khi query đơn giản và câu trả lời là thông tin công khai (giờ mở cửa, địa chỉ, thông tin chung về dịch vụ) → V-AI trả lời đúng, nhanh ✅ |
| **Low-confidence** | Khi query cần route/context cụ thể → V-AI không hỏi lại, trả luôn thông tin chung chung, không có clarification ⚠️ |
| **Failure** | Khi query cần dữ liệu thật của user (số dư, đơn hàng) → V-AI báo "không có quyền truy cập", hướng dẫn thủ công, không có fallback hữu ích ❌ |
| **Correction** | Khi user sửa hoặc cung cấp thêm context → V-AI không lưu correction, không học lại từ query fail, context biến mất sau mỗi session 🔴 |

```
User gõ/nói query
        │
        ▼
V-AI nhận input (voice/text)
        │
        ├─ [Happy path] ──────────────────────► Hiểu đúng intent → Trả lời ✅
        │
        ├─ [Low-confidence] ──────────────────► Trả thông tin chung chung ⚠️
        │                                        Không hỏi lại có context
        │
        ├─ [Cần data thật của user] ──────────► "Không có quyền truy cập" ❌
        │                                        Hướng dẫn thủ công
        │
        └─ [Cần thực hiện action] ────────────► "Liên hệ hotline/đại lý" 🔴
                                                 User tự xử, chờ 24–48h
```

**User bị kẹt nhiều nhất tại:** Bước "Cần data thật" và "Cần action" — V-AI chỉ là search engine chính sách, không kết nối được dữ liệu và hệ thống thật.

## 4. Viết finding thành quyết định

**Finding 1 — Data gap:**

```
Khi user hỏi thông tin cá nhân ("số dư ví", "đơn booking của tôi"),
V-AI không có quyền truy cập dữ liệu thật nên từ chối và hướng dẫn thủ công,
hậu quả là user phải tự tìm thông tin — V-AI không có giá trị gì so với việc tự vào app.
Lỗi thuộc layer data-tool: thiếu tích hợp API với hệ thống tài khoản/booking của Vingroup.
Nên sửa bằng: tích hợp read-only API để V-AI kéo được dữ liệu thật của user (sau khi xác thực),
hiển thị thông tin trực tiếp thay vì redirect.
```

**Finding 2 — Action gap + Time pressure (case ưu tiên):**

```
Khi user yêu cầu huỷ/đổi booking Vinpearl và hỏi về hoàn tiền,
V-AI trả về chính sách chung chung thay vì kéo đơn thật, tính toán theo context,
hậu quả là user không biết mình được hoàn bao nhiêu, deadline là khi nào,
và vẫn phải gọi hotline + chờ 24–48h — trong khi deadline hoàn tiền có thể hôm nay.
Lỗi thuộc layer data-tool + UX recovery: không có booking API, không có time alert, không có action button thật.
Nên sửa bằng: tích hợp Vinpearl booking API, hiển thị đơn thật + tính hoàn tiền chính xác,
thêm time alert deadline, button [Xác nhận huỷ] với confirm 2 bước,
và handoff có context đầy đủ cho agent nếu user cần hỗ trợ thêm.
```

**Finding 3 — Low-confidence không có clarification:**

```
Khi user hỏi route/giá cụ thể ("đặt xe từ Vinhomes đi Vinpearl Phú Quốc rẻ nhất"),
V-AI không hỏi lại để làm rõ context mà trả ngay thông tin chung,
hậu quả là user nhận được giá không liên quan đến yêu cầu thật, mất tin tưởng vào bot.
Lỗi thuộc layer intent + UX recovery.
Nên sửa bằng low-confidence path: detect khi query cần context cụ thể → hỏi lại
(điểm đón, thời gian, loại xe) trước khi trả kết quả.
```

## 5. Sketch as-is / to-be

### As-is (flow hiện tại)

```
User: "Huỷ 1 đêm đầu + đổi phòng, còn hoàn tiền không?"
        │
        ▼
V-AI xử lý text
        │
        ▼
Tìm trong knowledge base về chính sách Vinpearl
        │
        ▼
Trả về: "Theo chính sách, hoàn 50% nếu huỷ trước 7 ngày"  ⚠️ [Thông tin chung, không theo đơn thật]
        │
        ▼
Quick buttons: [Xem chính sách] [Liên hệ hotline] [Tìm phòng khác]  ❌ [Không có action thật]
        │
        ▼
User gọi hotline → chờ 24–48h → tự xử lý  🔴 [Time pressure: có thể hết deadline hoàn tiền]
```

**Điểm gãy:** Không có API booking → không có dữ liệu thật → không tính được → không action được → redirect sang hotline.

---

### To-be (flow đề xuất)

```
User: "Huỷ 1 đêm đầu + đổi phòng, còn hoàn tiền không?"
        │
        ▼
V-AI kéo booking thật của user (API Vinpearl — sau xác thực)
        │
        ▼
Hiển thị: Đơn #XXXX · Check-in 10/6 · 3 đêm · Phòng Deluxe
          ⚠️ Deadline hoàn tiền 50%: HÔM NAY 23:59 (còn X giờ)
        │
        ▼
V-AI hỏi lại: "Bạn muốn huỷ đêm nào? Phòng muốn đổi loại gì?"
        │
        ▼
Tính toán: Hoàn 50% đêm 1 = X.XXX VNĐ (3–5 ngày về tài khoản)
           Upgrade lên Deluxe Ocean View: chênh lệch X.XXX VNĐ/đêm
        │
        ├─ [Xác nhận huỷ đêm 1] → Confirm lần 2 → Thực hiện → Gửi email xác nhận ✅
        │
        └─ [Yêu cầu đổi phòng] → Connect agent Vinpearl với full context ✅
                                  (không cần user giải thích lại từ đầu)
```

**So sánh chi tiết As-Is vs To-Be:**

| Yếu tố | As-Is | To-Be |
|---|---|---|
| **Source dữ liệu** | Policy chung từ knowledge base | Kéo đơn booking thật qua API |
| **Tính toán hoàn tiền** | Không có | Hiển thị số tiền cụ thể + deadline |
| **Clarification** | Không hỏi lại | Xác nhận đêm huỷ + loại phòng muốn đổi |
| **Action button** | Generic (Xem chính sách, Hotline) | [Xác nhận huỷ] [Yêu cầu đổi phòng] |
| **Confirm trước action** | Không có | Confirm 2 bước trước khi huỷ thật |
| **Time alert** | Không có | Banner: "Còn X giờ để được hoàn 50%" |
| **Handoff sang agent** | "Gọi hotline, chờ 24–48h" | Connect thẳng với full context |
| **Correction log** | Không lưu | Ghi nhận query fail để cải thiện model |

## Product Decision

> **"V-AI hiện chỉ là search engine chính sách — biết thông tin nhưng không truy cập được dữ liệu thật của user và không thực hiện được action nào. Ưu tiên sửa: tích hợp booking API để V-AI đọc được đơn hàng của user, tính toán chính xác theo context thật (thời gian, loại phòng, chính sách áp dụng), và thực hiện action với confirmation 2 bước + time alert, thay vì redirect toàn bộ sang hotline và chờ 24–48h."**
